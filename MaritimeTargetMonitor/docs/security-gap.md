# 安全风险与兼容升级方案

## 1. 已知安全风险

| 风险项 | 描述 | 影响 | 替换方案 | 兼容性说明 |
|--------|------|------|----------|------------|
| HTTP协议 | 使用HTTP而非HTTPS | 数据传输可能被窃听 | 升级为HTTPS | 保持端口配置兼容，通过IsHttp字段控制 |
| MD5加密 | 密码使用MD5加密 | MD5已被证明不安全 | 使用SHA-256或更高强度算法 | 保持接口兼容，后端同时支持MD5和新算法 |
| 默认账号 | 可能存在默认账号 | 系统可能被未授权访问 | 移除默认账号，强制首次登录修改密码 | 保持登录接口兼容 |
| enableServer=false | 离线模式可能绕过认证 | 安全控制失效 | 添加本地认证机制 | 保持离线模式功能，增加本地密码保护 |
| 缺少请求超时处理 | 部分请求可能无限等待 | 系统稳定性风险 | 添加请求超时机制 | 保持接口兼容，增加超时配置 |
| 缺少输入验证 | 部分接口可能存在注入风险 | 系统安全风险 | 添加输入验证 | 保持接口兼容，增加服务器端验证 |

## 2. 安全加固建议

### 2.1 网络安全

- **启用HTTPS协议**：默认使用HTTPS，通过config.json中的IsHttp字段控制是否回退到HTTP
- **配置合理的超时和重试机制**：添加请求超时配置，避免无限等待
- **实现请求频率限制**：防止暴力破解和DoS攻击

### 2.2 认证与授权

- **使用强密码哈希算法**：升级为SHA-256或Argon2，同时保持对MD5的兼容支持
- **实现Token过期机制**：添加Token有效期设置，定期刷新
- **添加权限细粒度控制**：基于角色的访问控制，限制敏感操作

### 2.3 数据安全

- **对敏感数据进行加密存储**：加密存储用户密码和敏感配置
- **实现数据脱敏**：在日志和API响应中对敏感信息进行脱敏
- **定期备份数据**：建立数据备份策略，防止数据丢失

### 2.4 代码安全

- **进行代码安全审计**：定期检查代码中的安全漏洞
- **修复潜在的安全漏洞**：及时更新依赖库，修复已知漏洞
- **保持依赖库更新**：定期更新第三方依赖，避免使用有漏洞的版本

### 2.5 运维安全

- **定期更新系统和依赖**：保持操作系统和软件依赖的更新
- **配置合理的日志记录**：记录关键操作和安全事件，便于审计
- **实现监控和告警机制**：实时监控系统状态，发现异常及时告警

## 3. 兼容升级方案

### 3.1 HTTPS默认启用

- 修改config.json默认值：`"IsHttp": false`
- 提供证书配置选项，支持自签名证书和CA证书
- 保持对HTTP的兼容支持，确保旧系统可以正常运行

### 3.2 密码加密升级

- 后端同时支持MD5和SHA-256加密
- 当用户首次登录时，自动将密码哈希升级为SHA-256
- 前端保持密码加密逻辑不变，后端负责检测和升级

### 3.3 移除默认账号

- 移除系统中的默认账号
- 实现首次登录强制修改密码功能
- 保持登录接口的向后兼容性

### 3.4 离线模式安全增强

- 添加本地认证机制，即使在离线模式下也需要密码验证
- 本地密码与服务器密码同步，确保一致性
- 保持离线模式的核心功能不变

### 3.5 请求超时与输入验证

- 添加请求超时配置，默认值设置为合理范围
- 实现服务器端输入验证，防止注入攻击
- 保持API接口的向后兼容性，不影响现有客户端
